## 1. 產品概述

VeloSays 是一款**個人化跑步訓練儀表板**，自動整合 Garmin 手錶的跑步紀錄與每日體能數據，搭配 AI 教練（Gemini）提供訓練分析、聊天問答與每月訓練目標，協助使用者備戰目標賽事。

### 1.1 目標使用者與情境

- 單一使用者：Sasha（`user_id: c8f7c70c-7fbd-416d-8dbc-e817bf827e84`）
- 目標賽事：**2026-10-25 金澤全程馬拉松，目標 Sub 4:00**（設定於 `src/raceConfig.ts` 與 `scripts/sync_garmin.py` 的 `TARGET_RACE`）
- 使用裝置：以手機瀏覽器為主（430px 手機殼版型，桌面瀏覽時置中顯示）

### 1.2 核心價值

1. **零手動輸入**：跑步與體能數據每日自動從 Garmin 同步
2. **看得懂的數據**：把原始數據轉成訓練負荷（TSB）、80/20 強度比例等可操作指標
3. **AI 教練**：依真實訓練數據給建議的聊天教練＋每月自動生成訓練目標

## 2. 系統架構

```
┌─────────────┐   每日 06:00 台北     ┌──────────────┐
│   Garmin     │ ◄──────────────────  │ GitHub Actions│
│   Connect    │   garminconnect/garth │ sync_garmin.py│
└─────────────┘   (token 快取)        └──────┬───────┘
                                             │ service_role key
                                             ▼
                                      ┌──────────────┐
                                      │   Supabase    │
                                      │  (PostgreSQL) │
                                      └──────┬───────┘
                                             │ publishable key（唯讀）
                                             ▼
┌─────────────┐    strava_chat        ┌──────────────┐
│ Google Apps  │ ◄──────────────────  │ React 前端    │
│ Script + │    text/plain POST    │ (Vercel)      │
│ Gemini       │ ──────────────────►  │               │
└─────────────┘    教練回覆            └──────────────┘
```

| 元件 | 技術 | 職責 |
|---|---|---|
| 前端 | React 19 + TypeScript + Vite + Tailwind CSS 4 | 五分頁 UI、指標計算、讀取 Supabase |
| 資料庫 | Supabase（PostgreSQL + PostgREST） | activities / daily_metrics / monthly_goals 三張表 |
| 同步管線 | GitHub Actions + Python（garminconnect 0.2.8） | 每日拉 Garmin 資料寫入 Supabase；月底生成月目標 |
| AI 教練聊天 | Google Apps Script + Gemini（`gemini-flash-latest`） | 接收前端聊天請求，帶跑步紀錄 context 問 Gemini |
| 部署 | Vercel（GitHub 整合） | push main 即自動部署 |

## 3. 功能規格

### 3.1 看板（Dashboard）

| 區塊 | 內容 | 資料來源 |
|---|---|---|
| 頁首 | 日期＋「Hi, Sasha」＋重新整理鈕 | 重新整理＝重抓 Supabase（非觸發 Garmin 同步） |
| 可滑動日曆條 | 過去 90 天～未來 7 天，今日高亮置中；有跑步的日期顯示橘點，**點擊跳轉該筆詳情** | activities |
| 目標賽事卡 | 賽事名＋目標＋倒數天數 | `raceConfig.ts`（使用者提供的固定資訊） |
| 本月跑量卡 | 本月累積 km／AI 月目標 km、進度條、還差 km、教練 summary；無目標時顯示「將於月底自動生成」 | activities + monthly_goals |
| 統計卡 ×4 | 最近配速、平均心率、VO2max、睡眠分數 | activities（最新一筆）+ daily_metrics |
| 本週跑量長條圖 | 週一～週日每日 km | activities |
| 同步時間註記 | 「體能資料同步於 {date}」 | daily_metrics.date |

### 3.2 歷史（HistoryList）

- 三張統計卡：本月累積里程（藍漸層）、本月跑步次數（橘漸層）、累積總次數（白）
- 列表每頁 6 筆，分頁控制（‹ 1/N ›，邊界失效）
- 每筆：類型 icon（依類型配色）、日期＋類型、配速/心率/**時間（hh:mm:ss）**
- 點擊進入詳情；返回時保持分頁位置

### 3.3 跑步詳情（RunDetail）

- 副標依資料有無顯示：日期・城市・溫度・濕度（天氣欄位僅新同步的活動有值）
- 漸層 hero：距離／配速／時間（hh:mm:ss）
- 2×2 指標卡：平均心率、最高心率、平均步頻、平均步幅
- 條件顯示區塊（目前資料庫尚無資料，有值才渲染）：心率區間 Z1–Z5、分段配速、教練回饋
- 詳情頁時底部導覽維持「歷史」高亮

### 3.4 AI 教練（Coach）

- 聊天介面：教練白泡泡左對齊、使用者藍漸層右對齊、「教練思考中...」指示
- 快捷 chips：「訓練負荷分析」「心率區間」
- 送出時附帶**最近 10 筆真實跑步紀錄**作為 context
- 後端：GAS `strava_chat` action → Gemini `gemini-flash-latest`
  - **必要參數**：`thinkingConfig: { thinkingBudget: 0 }`（否則內部思考吃光 maxOutputTokens 導致回覆被腰斬）
  - Prompt 已調校為口語聊天風格（禁 ### 標題、禁條列、2-4 段）

### 3.5 體能狀態（FitnessStatus）

| 指標 | 定義 | 備註 |
|---|---|---|
| VO2max | Garmin 每日估算值 | 只在跑步日有值，同步時往回找 30 天取最新 |
| 訓練負荷狀態 TSB | CTL(42天EMA) − ATL(7天EMA)，TRIMP 用「心率儲備比例×分鐘×100」線性近似 | **簡化估算**，非實驗室精算；>+5 偏新鮮、<−10 疲勞偏高 |
| 低強度跑量比例 | 近 28 天，心率儲備 ≤75% 為低強度（80/20 法則參考） | 用單次平均心率分類，非逐秒區間 |
| 安靜心率／睡眠分數 | Garmin 每日數據 | daily_metrics |
| 7 天訓練負荷圖 | 每日運動分鐘數長條圖 | activities |

### 3.6 設定（Settings）

- 佔位頁（「此頁面尚未實作」）

## 4. 資料模型（Supabase）

### 4.1 `activities`（跑步紀錄）

| 欄位 | 型別 | 說明 |
|---|---|---|
| id | uuid PK | Garmin 活動以 `uuid5(NAMESPACE_URL, "garmin-activity-{activityId}")` 生成 → upsert 冪等 |
| user_id | uuid | |
| date | date | 活動當地日期 |
| distance | numeric | 公里 |
| pace | text | "MM:SS" 字串 |
| duration | int | **秒** |
| avg_hr / max_hr | int | bpm |
| avg_spm | int | 步頻 |
| avg_stride_length | int | **公分**（前端顯示時 ÷100 轉公尺） |
| source | text | `Garmin_Direct`（新）/ `Strava_Garmin`、`Garmin/Strava`（歷史遺留） |
| title | text | Garmin 活動名稱 |
| city / country | text | 天氣觀測站/城市（僅新同步活動） |
| temperature_c | numeric | °C（僅新同步活動） |
| humidity_percent | int | %（僅新同步活動） |

> 跑步類型（輕鬆跑/配速跑/長距離跑/恢復跑/間歇跑）**資料庫沒有欄位**，由前端 heuristic 推斷：title 關鍵字 → 距離 ≥15km 長距離 → 配速 <5:00 配速跑 → 距離 ≤4km 恢復跑 → 其餘輕鬆跑。

### 4.2 `daily_metrics`（每日體能）

| 欄位 | 型別 |
|---|---|
| user_id + date | 複合 PK |
| vo2max | numeric |
| resting_hr | int |
| sleep_score | int |

### 4.3 `monthly_goals`（AI 月目標）

| 欄位 | 型別 | 說明 |
|---|---|---|
| user_id + year_month | 複合 PK | year_month 格式 `YYYY-MM` |
| goal_km | numeric | AI 生成的月跑量目標 |
| summary | text | 教練一句話訓練重點 |
| focus | text | 重點關鍵字 |

三張表皆開 RLS：anon/publishable 可 select，寫入僅 service_role（由同步腳本持有）。

## 5. 同步管線規格（`scripts/sync_garmin.py`）

### 5.1 排程與觸發（2026-07-20 起改為本機優先）

- **主要排程：使用者 Mac 上的 launchd**，每日 **07:30 與 09:00 各執行一次** `scripts/local_sync.sh`（07:30 先抓跑步/心率；09:00 補當天睡眠分數，Garmin 通常此時已算好）
  - plist：`~/Library/LaunchAgents/com.velosays.garmin-sync.plist`
  - 秘鑰檔：`~/.velosays-sync.env`（chmod 600，不進 git）
  - log：`~/Library/Logs/velosays-sync.log`
  - 電腦睡眠時錯過的排程會在喚醒後補跑；關機錯過則當日不跑
- GitHub Actions：**每日 cron 已停用**（見下方事故記錄），保留 workflow_dispatch 手動觸發供解封測試
- 手動執行：`bash scripts/local_sync.sh` 或 `launchctl kickstart gui/$(id -u)/com.velosays.garmin-sync`

### 5.2 登入策略（依序）

1. **快取 session**（`~/.garminconnect`）：oauth2 未過期 → 直接用，**不打 OAuth2 換發端點**
2. 快取的 oauth2 過期但 oauth1（效期約一年）還在 → 用 oauth1 換發一次新 oauth2 並存回快取（**本機日常路徑**，不依賴 GARMINTOKENS）
3. `GARMINTOKENS` secret：載入後觸發一次換發（CI 用）
4. 本機帳密備援（僅限本機，雲端 IP 帳密登入必 429）

> **重要教訓（2026-07-15 ~ 07-20 事故）**：GARMINTOKENS 內的 oauth2 token 效期僅約 1 小時，過期就要重新換發；7/15 密集手動測試在單日對換發端點打了十多次，觸發 429。之後**連續 4 天、每天僅一次**的排程換發仍持續 429 → 判定 Garmin 已對 GitHub 機房 IP 段實施長期封鎖（帳號與家用 IP 均不受影響，本機實測即通）。因此每日排程改為本機 launchd 執行；**任何時候都不要在鎖定期間反覆重試**。

### 5.3 每次執行的工作

1. 拉當日體能指標（VO2max/安靜心率/睡眠分數）→ upsert `daily_metrics`
   - 所有日期以 **台北時區** 計算（`today_taipei()`；runner 是 UTC，直接用 `date.today()` 會整套錯一天）
   - **睡眠分數的時間差**：早上 07:30 執行時 Garmin 常還沒算好「今天」的睡眠分數，故排程改為 07:30＋09:00 各跑一次（09:00 通常已算好）；程式端另有保險——先試今天、抓不到退回抓昨天（睡眠分數本就是前一晚的睡眠品質，語意上對應昨天也正確）
2. 拉近 60 天跑步活動 → 去重後 upsert `activities`
   - 去重：同日期＋時長差<60秒＋距離差<0.3km 視為同一筆（涵蓋 Strava 時代舊資料）
   - 新活動同時呼叫 `get_activity_weather()` 補天氣欄位
3. 若為**當月最後一天**（或 `FORCE_MONTHLY_GOAL=1`）：彙整當月跑量＋近 35 天體能趨勢 → Gemini 生成下月 `goal_km`/`summary`/`focus` → upsert `monthly_goals`

### 5.4 GitHub Secrets

| Secret | 用途 | 效期/輪替 |
|---|---|---|
| `GARMINTOKENS` | Garmin OAuth token 字串 | oauth1 約 1 年；過期重跑 `garmin_login.py` |
| `SUPABASE_SERVICE_ROLE_KEY` | Supabase 寫入（sb_secret_ 新制 key） | 可於 Supabase Dashboard 個別輪替 |
| `GEMINI_API_KEY` | 月目標生成 | 與 GAS 共用同一把 |

## 6. 設計系統

- **配色**：冷藍 `#2563EB` × 活力橘 `#F97316`；文字 `#1C2430`/`#6A7482`/`#9AA3B0`；App 背景漸層 `linear-gradient(170deg, #E8F1FF, #F7F9FF 45%, #FFF0E2)`
- **字體**：中文 Noto Sans TC（400/500/700）、數字 Outfit（300–600）
- **版型**：430px 手機殼（`max-width: 430px` 置中、`h-dvh`、底部導覽毛玻璃固定）；圓角卡片 20–28px
- **圖示**：Tabler Icons webfont
- **關鍵 CSS 陷阱**：分頁是固定高度 flex column＋overflow-y-auto，子元素必須 `flex-shrink: 0`（index.css 已全域處理），否則內容超高時會被壓扁而非捲動
- 設計參考稿：`design_handoff_velosays_app/`

## 7. 已知限制與注意事項

| 項目 | 現況 |
|---|---|
| 天氣欄位對應 | 已用真實回應校正（2026-07-20）：`temp` 為**華氏**（腳本轉攝氏；已與 Garmin App 顯示的 27°C 交叉比對確認——App 介面顯示攝氏是 Garmin 轉換過的，API 原始值是華氏，勿混淆）、`relativeHumidity` %、`weatherStationDTO.name` 觀測站名；僅新同步的活動有值 |
| 心率區間/分段配速/教練回饋 | 資料庫無此資料，詳情頁對應區塊隱藏 |
| 跑步類型 | heuristic 推斷，非 Garmin 官方分類 |
| TSB / 80-20 | 簡化公式（見 3.5），供趨勢參考非醫療建議 |
| VDOT | 刻意未做——需要指標賽事成績當基準，現有資料算不準 |
| 設定頁 | 未實作 |
| garminconnect | 非官方套件（版本鎖 0.2.8，與 GARMINTOKENS 格式綁定），Garmin 改版可能需要更新 |
| GAS 舊部署 | 專案內有多個歷史部署，現役 URL 見 `src/api/activities.ts` 的 `SYNC_ENDPOINT`；改 GAS 程式後務必用「管理部署→新版本」更新**現役**部署 |
| 舊 Strava 路徑 | 已棄用（Strava API app 停用中），`src/api/trainingApi.ts` 為死碼可清除 |

## 8. 維運速查

```bash
# 手動跑一次同步（平常不用——launchd 每日 07:30 自動跑）
bash scripts/local_sync.sh

# 查看每日同步 log
tail -30 ~/Library/Logs/velosays-sync.log

# 排程管理
launchctl list | grep velosays                                   # 確認已註冊
launchctl kickstart gui/$(id -u)/com.velosays.garmin-sync        # 立刻跑一次
launchctl bootout gui/$(id -u)/com.velosays.garmin-sync          # 移除排程

# Garmin 重新登入（僅當 oauth1 過期，約一年一次）
GARMIN_EMAIL='...' GARMIN_PASSWORD='...' .venv/bin/python scripts/garmin_login.py
# → token 存 ~/.garminconnect（本機排程用）＋印出字串（GitHub secret GARMINTOKENS，備用）

# 強制重新生成當月 AI 目標
# （在 scripts/local_sync.sh 執行前 export FORCE_MONTHLY_GOAL=1，或：）
set -a; source ~/.velosays-sync.env; set +a
FORCE_MONTHLY_GOAL=1 .venv/bin/python scripts/sync_garmin.py

# 前端開發
npm run dev        # localhost:5173
npx tsc -p tsconfig.app.json --noEmit   # 型別檢查
npm run lint

# 部署：push main → Vercel 自動部署，無需手動操作
```
